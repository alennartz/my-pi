# Brief Template

Section structure for `docs/designs/<topic>.md`. Each decision states the call and a one-line rationale. Specify concrete values: hex codes, named font families, exact spacing units, named easing classes. No code snippets, no CSS, no framework-specific syntax.

Sections marked **(optional)** can be dropped when not relevant. Required sections are always present.

## Components

For each component the surface uses, specify these five things. Skip components the surface doesn't use.

1. **Anatomy** — the visual elements making it up.
2. **Variants** — meaningful variations needed (e.g., primary / secondary / ghost / destructive for buttons).
3. **States** — every state the component can be in.
4. **Behavior** — how it responds across input modes (pointer, keyboard, touch).
5. **Content rules** — voice, length limits, capitalization.

**State checklist** — every interactive component needs decisions for:

- Default / Hover (pointer) / Focus-visible / Active (pressed) / Disabled / Loading
- Selected (when it represents a chosen option)
- Empty / Error (for containers and content components)

Focus-visible is never optional. If the brief doesn't specify a state, the implementer will guess — usually generic-default badly. Specify states even briefly.

> Note for the implementer: this section describes intent and timing classes — not implementation. Translate to your platform's animation primitives honoring the timing classes and continuity rules.

**Motion philosophy.** <one or two sentences on motion's role on this surface>

**Timing classes.**
- *Snappy* (~100-150ms) — button press, immediate state changes
- *Standard* (~200-300ms) — dropdowns, popovers, page-internal transitions
- *Expressive* (~400-700ms) — modals, page transitions, marketing reveals *(only if used)*

**Where animation appears.**
- <list moments + timing class — e.g., "Button press: snappy / Tab change: standard / Hero entrance: expressive">

**Continuity rules.**
- Shared elements between views animate in place, not blink.
- Overlays emerge from their trigger element, not the void.
- Exit animations are faster than entrance.
- Keyboard-initiated actions do not animate.
- `prefers-reduced-motion: reduce` replaces all motion with instant state changes.
